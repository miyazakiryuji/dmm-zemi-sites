STEP 1

Git とは（バージョン管理）

ファイルの変更履歴を記録し、いつでも戻せる・共同作業できる仕組み

Git（ギット） は、プログラムやファイルの 変更履歴を記録する道具（バージョン管理システム）です。「いつ・誰が・どこを・なぜ変えたか」を残し、前の状態に戻したり、複数人で同じプロジェクトを安全に編集したり できます。

そもそも、なぜ Git が必要？

Git が無いと、変更を残すために 企画書_最新.txt／企画書_最新_v2.txt／企画書_本当に最終.txt … とコピーを増やしがちです。これだと どれが最新か分からない・前の状態に戻せない・他の人と同時に直すと上書き事故 が起きます。

Git は、この“手作業のコピー管理”を置き換えます。フォルダは1つのまま、節目ごとに記録（コミット）していくので、履歴が一本の線として残り、いつでも振り返れます。

  • Git なし：ファイルを名前でコピー → 最新が不明・戻せない・上書き事故
  • Git あり：1つのフォルダで履歴を記録 → 最新が明確・いつでも復元・安全に共同作業

ゲームの“セーブポイント”に近い

編集の区切りごとに コミット（commit） という“セーブ”を作ります。失敗しても直前のコミットに戻せるので、安心して大胆に変更できます。

Git は 分散型。手元のパソコンに履歴の“まるごとコピー”を持つので、ネットが無くても記録でき、後でまとめて共有できます。

コミットは“セーブポイント”として一列に残る

C1最初の commit
C2機能を追加
C3文言を修正HEAD（今ここ）

いつでも前のコミットに戻れるので、安心して試せます。

Git のメリット

まとめると、こんな“うれしさ”があります。

  • いつでも戻せる：壊しても前のコミットへ復帰できる。だから安心して大胆に試せる
  • 理由つきで履歴が残る：「いつ・誰が・なぜ変えたか」を後から追える
  • 安全に実験できる：ブランチで本線(main)を壊さずにお試しできる
  • 共同作業できる：複数人の変更を突き合わせて1つにまとめられる
  • バックアップになる：GitHub に置けば、パソコンが壊れてもコードは無事
  • 公開・デプロイの土台：多くの公開サービスが Git / GitHub 連携を前提にしている（EP24）

💡 本ゼミでは、こうした操作の多くを Claude Code に任せられます（最後の項目）。ただし“何が起きているか”を知っておくと、トラブル時に落ち着いて対処できます。

STEP 2

Git と GitHub の違い

Git＝手元で履歴を管理する“道具”／GitHub＝ネット上に置く“保管・共有の場所”

よく混同されますが、別物です。Git はあなたのパソコンの中で履歴を管理する ソフト。GitHub はその履歴（リポジトリ）を インターネット上に置いて共有・公開・バックアップ するための サービス（置き場所） です。

ざっくり対比

GitGitHub
正体バージョン管理ソフトGit リポジトリのホスティングサービス
場所手元のパソコン（ローカル）インターネット上（リモート）
役割変更を記録・管理する共有・公開・バックアップ・共同作業
無くてもGitHub 無しでも Git は使えるGitHub は Git があって成り立つ

GitHub の代わりに GitLab・Bitbucket などもありますが、本ゼミでは GitHub を使います。

ローカル（Git）とリモート（GitHub）の関係

💻
あなたのPC（ローカル）Git で履歴を記録（commit）
git push →
← git pull / clone
☁️
GitHub（リモート）ネット上で共有・公開・バックアップ

push でアップロード、pull / clone でダウンロード。Git は手元、GitHub はネット上。

ℹ️ コースでは、作ったアプリの公開（Streamlit Community Cloud など）が GitHub 上のコードを読み込んでビルドする 仕組みのため、まず GitHub にコードを置きます。（EP24）

STEP 3

知っておく基本用語

この5語と“3つのエリア”が分かれば、ほぼ読めます

最初に、頻出する言葉だけ押さえましょう。

基本の5語

  • リポジトリ（repository / repo）：プロジェクト1つ分の入れ物。履歴ぜんぶが入っているフォルダ
  • コミット（commit）：ある時点の変更を1つの“記録”として確定したもの（＝セーブ）
  • ブランチ（branch）：作業を枝分かれさせる線。ふつう本線は main
  • リモート（remote）：ネット上のリポジトリ。GitHub 側のことを通常 origin と呼ぶ
  • HEAD：いま自分がいるコミット／ブランチを指す“現在地”の目印

変更が記録されるまで（コマンドはこの順で流れる）

作業ツリー編集中のファイル
git add→
ステージコミット候補
git commit→
ローカルリポジトリ履歴に確定
git push→
リモート(GitHub)ネットへ共有

💡 add → commit → push の3手が基本の流れ。まず「ステージに載せて」「記録して」「ネットに上げる」と覚えればOKです。

STEP 4

準備：インストールと初期設定

Git を入れて、名前とメールを一度だけ設定する

Git 本体を用意し、コミットに残す名前・メールを設定します。Mac は標準で Git が入っていることが多く、Windows は別途インストールが必要です。（EP25）

① Git があるか確認

ターミナルで次を実行。バージョンが表示されれば入っています。

git --version
  • Windows で「git が見つからない」と出たら、「Git for Windows」を公式サイトからインストール（エディタは閉じてから／終わったら再起動）。（EP25）
  • Mac で無い場合は Homebrew で brew install git。（EP25）

② 名前とメールを設定（最初の一度だけ）

コミットの“署名”になります。GitHub で使うメールにしておくと分かりやすいです。

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

ℹ️ ここで設定したメールは、Public リポジトリではコミット履歴から第三者に見えます。公開したくない場合は、GitHub の noreply メール（Settings → Emails で確認できる 〇〇@users.noreply.github.com）を user.email に設定してください。

③ GitHub アカウントと gh CLI

GitHub にコードを置くにはアカウントが必要です（Google 連携が手早い）。（EP24）

ターミナルから GitHub を操作するために GitHub CLI（gh） を入れて認証します。Claude Code 内では対話入力が必要なコマンドは先頭に ! を付けて、ターミナルへ直接流します。（EP25）

!gh auth login

! は「Claude Code を介さず、ターミナルでそのまま実行」の意味。リンクを開いて認証→表示コードを貼り付け→Authorize で完了。

STEP 5

基本サイクル：status → add → commit → push

毎回これの繰り返し。まず体で覚える4手

日々の作業は、ほぼこの流れの繰り返しです。

基本の4手＋戻すための2手

git statusいま何が変わっているか・ステージ済みかを確認（迷ったらまずこれ）
git add <ファイル>変更をステージに載せる。全部なら git add . もよく使う
git commit -m "変更内容の説明"ステージした変更を履歴に確定。メッセージは何をしたか短く
git pushローカルのコミットを GitHub（リモート）へ送る
git pullリモートの最新を取り込む（複数人・複数PCのとき）
git log --onelineコミット履歴を1行ずつ一覧で見る

例：ファイルを直して GitHub に反映するまで

zsh — myapp
# app.py を編集したあと…
$ git status
modified: app.py
$ git add app.py
$ git commit -m "翻訳ボタンを追加"
[main a1b2c3d] 翻訳ボタンを追加
$ git push
✓ GitHub に反映されました

status で確認 → add → commit → push。この4手の繰り返しが基本です。

はじめてのリポジトリを GitHub に上げる

新規プロジェクトを GitHub に置く最短の流れです。

git init -b main            # リポジトリを作成（最初のブランチ名を main に）
# ← この時点で先に .gitignore を用意（.env など秘密ファイルを必ず書く）
git add .                  # カレント配下の変更をまとめてステージ（削除も含む）
git commit -m "first commit"
gh repo create             # GitHub 側にリポジトリを作成（対話で名前・公開範囲を選ぶ）
git push -u origin main    # 送信＋以後の push 先を main に紐付け

本ゼミでは、この一連を Claude Code に「GitHub にアップして」と頼むだけで自動実行できます（裏でこれらが走る）。（EP25）

⚠️ 先に .gitignore を作る（重要）

  • git add . / git add -A を .gitignore を作る前に 使うと、.env などの秘密ファイルごとステージしてしまいます。新規プロジェクトでは 必ず先に .gitignore を作成 してください（一度 push すると履歴に残り、実質取り消せません）。詳しくは項目8へ。

⚠️ git push が rejected（拒否） されたら、たいていリモートに自分の手元に無い変更があります。先に git pull で取り込んでから push し直します。

STEP 6

もう少し詳しく：主要コマンド

使う頻度の高いものだけ、目的とセットで

丸暗記は不要。「やりたいこと」から引けるように並べました。

確認・記録

git status変更・ステージ状況の確認
git diffまだステージしていない変更の中身を表示
git diff --stagedステージ済みの変更の中身を表示
git add -Aリポジトリ全体（いま居る階層の外も含む）を対象に全変更をステージ
git commit -m "..."履歴に確定
git log --oneline --graph履歴を簡潔＋枝分かれ付きで表示

取り消し・戻す（やさしめ）

git restore <ファイル>まだコミットしていない編集を捨てて、最後のコミット状態に戻す
git restore --staged <ファイル>ステージから降ろす（変更自体は残す）
git revert <コミット>指定コミットを“打ち消す”新しいコミットを作る（履歴は消さず安全）

リモート

git remote -v連携先（origin など）のURLを確認
git remote add origin <URL>リモートを手動で登録
git clone <URL>GitHub 上のリポジトリを手元へ丸ごと複製
git fetchリモートの更新を取得（まだ手元には反映しない）
git pull取得＋手元へ反映（fetch + merge）

ℹ️ reset --hard など“履歴ごと消す”系は強力な反面、戻せない事故になりがちです。まずは上の restore / revert を使い、迷ったら Claude Code に「安全に取り消したい」と相談しましょう。

STEP 7

GitHub と連携する

リモート作成・push/pull・clone・Public/Private

手元のリポジトリを GitHub につなぎ、共有・公開します。

GitHub 連携でよく使う

gh repo createGitHub 側に新しいリポジトリを作る（対話で公開範囲を選択）
gh repo create --private最初から非公開（Private）で作成
git push -u origin main初回の送信＋追跡ブランチを設定（次回以降は git push だけ）
git pullGitHub 側の最新を取り込む

Public と Private

Public は誰でも閲覧可能、Private は自分（と許可した人）だけ。作成時に公開範囲を選びますが、初期選択が Public 寄りで見落としやすい ため、個人プロジェクトは Private を推奨 します。（EP25）

  • あとから変更：GitHub → リポジトリの Settings → 一番下の「Change repository visibility」で切替
  • 最初から非公開：gh repo create --private、または Claude Code に「Private でアップして」と頼む

ℹ️ 公開サービスによっては制約があります。例：Streamlit Community Cloud の無料プランは Public リポジトリのみ デプロイ可。（EP26）

STEP 8

.gitignore と機密情報

APIキーなどの“秘密”を、絶対に GitHub に上げない

.gitignore は「Git で追跡しない（アップしない）ファイル」を書いておく設定ファイルです。API キーやパスワードの流出を防ぐ最重要ポイント です。（EP25）

.gitignore の書き方

リポジトリのルートに .gitignore を置き、除外したいものを1行ずつ書きます。

.env
*.pyc
__pycache__/
node_modules/
.DS_Store

.env は API キーなどの秘密を入れるファイル。.env.example（中身は空のテンプレ）だけ共有するのが定番。

⚠️ もし秘密を push してしまったら

  • まず .gitignore に .env を追加し、追跡から外す：
  • API キーは“流出した前提”で即無効化 → 新しいキーを再発行（最優先）。（EP25）
git rm --cached .env
git commit -m "Remove .env from tracking"
git push

--cached は手元のファイルは消さず、Git の追跡からのみ外す指定。※過去のコミットにも残るため、本当に消すならキー無効化が必須。

STEP 9

ブランチの基礎

本線(main)を壊さずに、別の線で試す

ブランチ は作業を枝分かれさせる仕組み。新機能やお試しを main とは別の線 で進め、うまくいったら main に合流（マージ）します。

ブランチ操作

git branchブランチ一覧と現在地を表示
git switch -c featurefeature という新ブランチを作って移動（旧: git checkout -b）
git switch mainmain へ戻る
git merge feature（main にいる状態で）feature の変更を取り込む

ブランチの分岐とマージ（図解）

git switch -c feature で枝を作り、作業して、git merge feature で main に合流。

  • 同じ箇所を別ブランチで違う内容に変えると コンフリクト（衝突） が起きることがあります。あわてず、残す内容を選んで解決します（Claude Code に解消を手伝ってもらうのも有効）。
  • 最初のブランチ名が master になっていて GitHub の main と食い違うことがあります（EP26 でも、Deploy 時の URL のブランチが main でなく master の場合がある旨に触れています）。揃えるには git branch -M main が使えます。

STEP 10

Claude Code に Git を任せる（コースの流儀）

「GitHubにアップして」で、裏側の git を自動実行

本ゼミの実践では、Git 操作の多くを 日本語の指示で Claude Code に任せられます。コマンドを完璧に覚えていなくても進められるのが強みです。（EP25）

たとえばこう頼む

こう指示すると、裏で git init・git add・git commit・gh repo create などが実行されます。（EP25）

この生成 AI アプリケーションを GitHub のリポジトリにアップロードしてください— EP25 の指示例（Private にしたいときは「Private で」と添える）

任せられること（例）

GitHubにアップしてinit〜commit〜リポジトリ作成〜push まで一括
わかりやすいコミットメッセージで記録して変更内容を要約したメッセージで commit
.gitignore に .env を追加して機密ファイルの除外設定を作成・修正
安全に1つ前の状態へ戻したいrestore / revert など安全な手段を提案・実行

⚠️ gh auth login のように 対話入力が必要なコマンド は Claude Code 内だと固まることがあります。その場合は先頭に ! を付けて !gh auth login のようにターミナルへ直接流します。（EP25）

💡 任せきりでも進みますが、status / log / push / pull / .gitignore の意味 だけ分かっていれば、想定外の挙動にも落ち着いて対処できます。

FAQ

よくある質問・トラブル

つまずきやすいポイントと対処法。

Windows で「git が見つからない」と出ます
Windows は Git が標準で入っていません。公式サイトから「Git for Windows」をインストールし（エディタは閉じてから／終わったら再起動）、git --version で確認してください。Mac は標準で入っていることが多く、無ければ brew install git。（EP25）
.env（APIキー）を GitHub に上げてしまいました
まず .gitignore に .env を追加し、git rm --cached .env → コミット → push で追跡から外します。ただし過去のコミットには残るため、漏れた API キーは流出した前提で即無効化し、新しいキーを再発行してください（こちらが最優先）。（EP25）
Private にしたかったのに Public で作られました
作成時に公開範囲の選択を見落とし、Public のまま進んでしまうことが原因です。リポジトリの Settings → 一番下「Change repository visibility」で Private に変更できます。最初から非公開にするには gh repo create --private、または Claude Code に「Private でアップして」と頼みます。（EP25）
git push が拒否（rejected）されます
多くは、リモート（GitHub）に自分の手元へ未取得の変更があるためです。先に git pull で取り込み、必要なら衝突を解消してから、もう一度 push します。
コマンドを全部覚えないとダメ？
いいえ。本ゼミでは Git 操作の多くを Claude Code に日本語で任せられます。まずは status / add / commit / push / pull の流れと .gitignore の意味だけ押さえれば十分です。（EP25）
