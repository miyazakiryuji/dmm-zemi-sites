プロンプトエンジニアリング応用編の課題から学ぶ
「伝わるプロンプト」の作り方

Lesson4 Chapter4 の3つの提出課題（変数 / コマンド / 条件分岐）について、提出された解答を匿名で抜粋しました。 「良かった提出」と「あと一歩の提出」を見比べながら、教材の評価観点に沿って“どこを直せばより伝わるプロンプトになるか”を整理します。

対象：Lesson4 / Chapter4 課題数：3 テクニック レビュー対象：提出 11 件 提出形態：役割・命令・文脈・制約・出力指示の型を活用 ※氏名・ハンドル名は伏せ、内容のみを抜粋しています
01

課題の概要

Chapter4「応用編」では、入門・基本編で学んだプロンプトの基本5要素を土台に、3つの上級テクニックを実務テーマで使う課題が出されました。

提出課題①

変数を使う

毎回変わる情報を { } の変数として切り出し、プロンプトをテンプレート化する。普段の業務で繰り返し使える形にするのがねらい。

提出課題②

コマンドを使う

[C1][C2]… と実行プロセスを順番に指示する。各手順が具体的で、前の結果が次につながる「流れ」を設計する。

提出課題③

条件分岐を使う

ニュース記事が自社にとってポジティブか否かを判定させ、判定結果によって異なるアクションを指示する。例外処理を含む実務的な分岐を作る。

共通テーマ（課題③）： 「児童向けプログラミング教育サービス」を自社と想定し、『8割の小学生保護者が学校以外でプログラミングを学ばせていない』という調査記事を分析。判定に応じて次の一手を提案するプロンプトを設計する、という同一お題で全員が取り組みました。
02

評価の観点

教材（Chapter2〜4）で示された基準に沿ってレビューしています。まずは全課題に共通する“土台”、続いて各テクニック固有のチェックポイントです。

土台：プロンプトの基本5要素（Chapter2・3）

① 役割

AIに何者として振る舞ってほしいか（例：情報セキュリティ推進者、コピーライター）。

② 命令

実行してほしい具体的なタスク（例：比較表を作る、告知文を書く）。

③ 文脈

背景・状況。なぜ・誰のために行うのかを補う情報。

④ 制約事項

守るべき条件（字数、使ってはいけない表現、匿名化ルールなど）。

⑤ 出力指示

出力の形式・構造（表、箇条書き、◎○△評価、項目立てなど）。

＋ 構造化

# や * で見出し・箇条書きにし、テンプレートとして再利用できる形に。

テクニック別チェックポイント

テクニック「良い」と評価したポイントつまずきやすいポイント
① 変数 毎回変わる情報を過不足なく変数化／変数名が分かりやすい／値を入れて“動く状態”で提出／出力指示まで具体的 用途が私的で業務テンプレにならない／変数の値が未記入でAI任せ／出力形式の指定が「テキスト形式」だけ
② コマンド 手順が順序立っている／各コマンドが具体的／前の結果を次で使う“連鎖”を明記／出力形式や最終確認まで設計 手順が独立した箇条書きの羅列／判断基準を後出し／各ステップの出力形式が無い
③ 条件分岐 判定基準が具体的（数値・条件）／各分岐のタスクが具体／「該当分岐のみ実行」と指示／分岐ごとに出力形式を指定／最新情報を参照させる ポジ/ネガの判定基準が主観任せ／分岐はあるが出力形式が無い／自然文で1回に詰め込み再利用しにくい
良かった提出（抜粋） あと一歩・改善したい提出（抜粋）
03

提出課題①：変数

「毎回変わる情報」を変数に切り出し、テンプレートとして再利用できるか。値まで入れて実際に動く状態かが分かれ目でした。

GOOD 01 役員向け「セキュリティ製品 比較表」テンプレート

業務シーン：情報システム部門が、現行ツールと他製品を比較して役員に提案する場面。

# 役割 企業の情報セキュリティ推進者
# 命令 ・{service}の製品比較一覧を作成／説明相手はセキュリティ部門長
  ・役員が30秒で理解できる粒度／「推奨案」を必ず明示／現行製品を基準列に固定
# 補足情報 利用人数は{company_employee}人、現在は{main_product}を利用
# 出力指示 ・観点（コスト/セキュリティ/監査/利用しやすさ/移行）に従う
  ・◎○△で評価／説明文は1製品2行以内／結論を最後に1行
# 変数
{service}：セキュアなファイル共有ソフト
{company_employee}：2,000人
{main_product}：Box(Enterprise版)
✓ ここが良い
  • 「毎回変わる情報」を的確に変数化：比較したい製品ジャンル・利用人数・現行ツールを変数にし、別案件でも値を差し替えるだけで使える。
  • 出力指示が具体的：「◎○△評価」「1製品2行以内」「推奨案を明示」まで決めているため、誰が実行しても出力がブレない。
  • 役割・命令・補足情報・出力指示と5要素が揃い、そのままチームで共有できるテンプレートになっている。
GOOD 02 イベント告知文ジェネレーター（文章の「雰囲気」も変数化）

業務シーン：毎回テーマの異なる講座・イベントの告知文を、SNS用に量産する場面。

# 変数
- イベント名：{event_title}／対象者：{target_audience}／テーマ：{main_theme}
- 読者の悩み：{reader_problem}／参加後の未来：{desired_future}
- 開催日：{event_date}／形式：{event_format}／特典：{benefits}
- 申込案内：{call_to_action}／{tone}：文章の雰囲気
# 使用例（値を差し替えて再利用）
{event_title}：はじめてのマインドフルネス体験会
{tone}：やさしく、前向きで、安心感のある雰囲気 …
✓ ここが良い
  • 網羅性が高い：テーマ・対象・悩み・未来・特典・申込まで、告知文に必要な可変要素をひと通り変数化している。
  • 「文章のトーン」まで変数化：{tone}を用意し、同じ枠組みで“やさしい告知”も“きりっとした告知”も作り分けられる。
  • 「使用例」を添付：変数を差し替えた具体像を示しており、テンプレートが“動く状態”で提出されている。
あと一歩 手元の食材から料理を提案する

業務シーン：家にある食材を入力して献立を決める（私的な用途）。

# 役割 夕飯を毎日作る人
# 命令 材料をもとに作る料理を決めて手順を書き出す
# 変数
{material_1}: じゃがいも　{material_2}: 人参　{material_3}: とまと
# 出力指示 テキスト形式
△ 惜しいポイント
  • 役割・命令・変数の型は押さえている（基礎はOK）。一方で、課題のねらい「普段の業務で活用できる」テンプレートからはやや離れた題材になっている。
  • 出力指示が「テキスト形式」のみで、粒度の指定が無いため、出力の見た目が毎回変わりやすい。
Before の発想

「家にある食材」という1回限りの情報を変数にしている。

After（こう直すと伸びる）

「毎週の献立メール作成」など繰り返す業務を題材に、{宛先}{人数}{予算}{調理時間}{トーン}を変数化。出力指示に「見出し＋表形式・200字以内」を加えると、そのまま業務テンプレになる。

もう一つの“あと一歩”： 変数の定義（テンプレート）は秀逸なのに、肝心の値を入れずに提出したケースもありました。値が空のままだとAIが勝手に例を補完してしまいます。必ず1セットは具体値を入れ、「実行して結果が出る状態」で提出しましょう。
04

提出課題②：コマンド

手順を [C1][C2]… と分解できているか。さらに「前のコマンドの結果を次でどう使うか」まで設計できると、出力が一気に安定します。

GOOD 01 各コマンドの“受け渡し”を明記した告知文づくり

業務シーン：悩み→未来→内容→安心材料、と整理してから告知文に仕上げる。

[C1] 読者の悩みを整理 → ここで整理した悩みは[C5]の「冒頭文」「共感」に必ず反映
[C2] 望む未来を整理 → [C5]の「得られる未来」に反映
[C3] イベント内容を整理／[C4] 安心して参加できる要素を整理
[C5] 告知文の構成を作成（C1〜C4をどこに使うか意識）
[C6] 告知文を作成（C1〜C4の整理を必ず反映）
[C7] 文章の雰囲気を調整
[C8] 最終確認：条件を満たさなければ修正してから出力
✓ ここが良い
  • コマンドが“連鎖”している：各手順の末尾に「整理した内容を次のどの工程で使うか」を書いており、手順が分断されない。
  • 自己チェック工程まで設計：最後の [C8] で「条件を満たさなければ修正してから出力」と、教材の“自己改良”を組み込んでいる。
GOOD 02 「収集→匿名化→平易化→整形」と段階を積み上げる

業務シーン：社内向けに、今週のセキュリティ事例を注意喚起としてまとめる。

[C1] 客観的事実(Fact)を収集・整理する
[C2] 該当事例を絞り込み、被害組織を匿名化（業種・規模に変換）
[C3] 専門用語にカッコ書き説明を付け平易化（例：MFA（多要素認証））
[C4] マークダウンの表形式で出力
✓ ここが良い
  • 各コマンドが具体的で、次につながる：「集める→加工する→整える」と段階的に積み上がる設計。
  • 制約と出力形式を各手順に内蔵：匿名化ルール・用語注釈・表形式まで指示しており、出力がそのまま社内資料になる。
別解として光った形： /整理 → /原因分析 → /改善提案 → /優先順位 → /実行計画 のようにスラッシュ風コマンドで“次に渡す”流れを作った提出もありました。記法は自由でも「順番に積み上げる」発想が共通しています。
あと一歩 評価の「基準」を手順の後ろに置いてしまった例

業務シーン：接客ロールプレイングを評価し、良かった点・課題点をまとめる。

[C1] ①の場面を「良かった点／課題点」に600字でまとめる
[C2] ②の場面を「良かった点／課題点」に600字でまとめる
[C3] [C1][C2]を600字以内にまとめる
[C4] 良かった点の基準は「共感」「寄り添い」を優先 ← 基準が後出し
[C5] 課題点の基準は「共感欠如」「代案なし」を優先 ← 基準が後出し
△ 惜しいポイント
  • 「評価基準を明示する」という発想はとても良い。ただし基準（C4・C5）が、実際に評価を行う手順（C1〜C3）の後ろに来ている。
  • AIは上から順に読むため、基準が後出しだと前半の手順に基準が反映されにくい。
Before

評価する → そのあとで評価基準を提示。

After（順番を入れ替える）

基準を先に定義（[C1]良かった点の基準／[C2]課題点の基準）→ その基準で各場面を評価（[C3][C4]）→ まとめる（[C5]）。基準を手順の前に置くと、AIが一貫して同じ物差しで評価できる。

05

提出課題③：条件分岐

最も差がついた課題。「何をもってポジ/ネガと判定するか」の基準と、分岐ごとの出力形式を決められたかが評価の分かれ目でした。

GOOD 01 判定を「数値スコア化」して IF/ELSE で分岐

工夫：「ポジ/ネガ」を主観で決めさせず、市場データの条件でスコア1〜5に落とし込んでから分岐。

### 判定ルール（スコア1〜5）
【5:非常にポジティブ】市場浸透率10%未満 かつ 自社の強みを求める声50%超
【4:ややポジティブ】未実施50%以上 かつ その理由を自社の強みで解決できる
【3:中立】… 【2:ややネガ】… 【1:非常にネガ】…
### 条件分岐（IF / ELSE）
IF スコア4以上 → 事業戦略を3つ（各「実現性」を必須記載）
ELSE スコア3以下 → 改善点を3つ（各「課題/提案/実現性/タスク」）
### 出力形式 総括 → 比較表 → 提案3案（マークダウン）
✓ ここが良い
  • 判定基準が数値・条件で具体化：「なんとなくポジティブ」を排し、誰が見ても同じ判定になる。
  • 分岐ごとに出力項目を固定：4以上と3以下で書かせる内容を変え、出力形式まで指定している。
GOOD 02 ポジ/ネガ/中立の3分岐＋「該当分岐のみ実行」

工夫：2択でなく「中立」も用意し、暴走を防ぐ制約と、経営者向けの最終まとめ表まで設計。

### 影響判定 ポジティブ / ネガティブ / 中立 を判定（理由も）
### 条件分岐
条件A（ポジ） 事業機会／訴求メッセージ／SNS投稿案／NextAction3つ
条件B（ネガ） リスク／利用しない理由の仮説／改善案 …
条件C（中立） 示唆／注視点／発信の切り口 …
### 最終提案 表形式（判定結果/理由/示唆/優先アクション/実行内容）
出力条件：判定結果に該当するアクションのみ実行する
✓ ここが良い
  • 3分岐で“どちらとも言えない”を吸収：現実のニュース分析に近い、堅い設計。
  • 「該当分岐のみ実行」と明示：全分岐を出力する暴走を防止。「自社に都合よく解釈しすぎない」と恣意的判定を戒める制約を入れた提出も見られました。
  • 最後に経営者向けの表でまとめる出力指示があり、そのまま意思決定に使える。
あと一歩 自然文で1回に詰め込み、判定基準と出力形式が未指定

提出の傾向：依頼内容は的確で出力結果も良質。ただしプロンプトが会話文に近く、再利用しにくい。

下記URLの内容を分析し、プログラミングを学ばせたい保護者が
多いのか少ないのかを総合評価して。
学ばせたい保護者が多い場合は、なぜ通わせていないのかを分析して改善策を、
学ばせたくない保護者が多い場合は理由をまとめて。
#参照URL https://… （記事を参照させる工夫はGOOD）
△ 惜しいポイント
  • 記事URLを参照させている点は良い。一方で「多い/少ない」を判定する基準が示されておらず、判定がAIの主観任せになる。
  • 各分岐の出力形式（項目・字数・表）が無いため、出力の構造が安定しない。自然文1回の依頼で、テンプレートとして再利用しづらい。
Before

会話文で「多い/少ない→それぞれ◯◯して」と一気に依頼。

After（構造化する）

#役割 / #判定基準 / #条件分岐 / #出力形式 に分け、判定基準を具体化（例：未実施が50%以上ならポジ）。各分岐の出力項目を固定すれば、GOOD例のスコア型やA/B/C型に近づく。

06

全体の総括

11件を通して見えた、伸びている提出の共通点と、あと一歩の提出に共通する“つまずき”です。

伸びている提出の共通点

「出力の形」まで指示している

役割・命令で止めず、◎○△評価・表形式・字数・項目立てまで指定。AIに考えさせる前に“ゴールの見た目”を共有できている。

判断の「基準」を自分で決めている

条件分岐では、数値や条件で判定ルールを定義。コマンドでは評価基準を手順の前に提示。AIに丸投げしていない。

手順が「連鎖」している

各コマンドで「次にどう使うか」を明記。最後に自己チェックを入れる提出もあり、教材の自己改良を実践できている。

あと一歩の提出に多い点

出力指示が「テキスト形式」だけ

何を・どんな粒度で出すかが無く、出力の見た目が毎回ブレる。形式の指定を足すだけで安定する。

判定・基準がAI任せ

「ポジ/ネガ」を分けるが、その線引きの基準が無い。基準を1〜2行足すと一気に再現性が上がる。

自然文で1回に詰め込む

結果は良くてもテンプレート化できず、次に使い回せない。#で構造化し、変数・分岐を切り出す。

レビューを通じた一番の学び： 良い提出ほど「AIに考えさせる量」を減らし、判定基準・出力形式・手順の受け渡しを“人間側で先に決めている”傾向がありました。これはまさに Chapter4 が目指す「実行の手軽さ」と「意図した回答生成」の両立です。次の一歩は、自分の良い提出を変数化してチームで共有できるテンプレートに育てることです。
07

提出前セルフチェックリスト

次の課題や実務でプロンプトを書くとき、提出前にこの観点で見直すと“あと一歩”を埋められます。

共通（基本5要素）

  • ✓役割・命令・文脈・制約・出力指示の5要素がそろっているか
  • ✓# や * で構造化され、見出しで役割が一目で分かるか
  • ✓出力指示が「形式・粒度・字数」まで具体的か（「テキスト形式」だけになっていないか）

① 変数

  • ✓毎回変わる情報だけを変数にしているか（固定情報は本文に書く）
  • ✓変数に具体的な値を入れ、実行して結果が出る状態で提出しているか
  • ✓その題材は業務で繰り返し使えるテンプレートになっているか

② コマンド

  • ✓手順が [C1][C2]… と順序立てられているか
  • ✓各コマンドが前の結果を次で使う“連鎖”になっているか（独立した羅列でないか）
  • ✓評価基準は手順の前に置いているか／最後に自己チェック工程があるか

③ 条件分岐

  • ✓判定の基準（数値・条件）を自分で定義しているか（AI任せでないか）
  • ✓各分岐の出力項目・形式を固定し、「該当分岐のみ実行」と指示しているか
  • ✓最新情報はURL等でAIに参照させているか
