EP11 ・ 座学

Skillsで Claude Code に
“専門スキル”を追加する

Skills は「特定のタスク用の専門的な手順書・知識」をパッケージ化して Claude Code に追加できる拡張機能です。 このページは EP11『Skillsの基礎』の要点を、いつでも見返せるようにまとめた自習用の補助資料です。

What is Skills

そもそも Skills とは？

Claude Code は Skills なしの“単体”でも十分に活躍してくれます。そこに Skills を加えると、特定分野に特化した Claude Code を作ったり、業務ワークフローそのものを定義したりできるようになります。

Skills とは 「Claude Code に専門的な手順書・知識を追加できる拡張パッケージ」。
たとえば「会議の文字起こし → Notion データベースに保存」のような複数ステップの定型業務も、ひとつの手順としてまとめて一気通貫で実行させられます。

イメージは、先輩エンジニアが「このプロジェクト専用の作業マニュアル」を書いて渡してくれるような感じ。 そのマニュアルを読んだ Claude Code は、その分野の“できる人”として動いてくれます。

What You Can Do

Skills を入れるとできること

EP11 で紹介される、Skills を導入する主なメリットです。

特定分野に特化させる

フロントエンドのデザイン、セキュリティ監査など、専門スキルを持った Claude Code を作れます。汎用の状態より、その領域で的確に動いてくれます。

業務ワークフローを自動化

複数ステップの定型業務を、ひとつの手順として定義できます。例：文字起こし → 整形 → Notion に保存 を一気通貫で。

再利用・チームで共有

一度つくれば、いつでも呼び出して再利用できます。チームで共有すれば、プロジェクト固有のルールや手順をみんなで統一できます。

つまり Skills は 「この分野ならこのやり方」という“知識のかたまり”を Claude Code に持たせる仕組み。 手順を書いた本を一冊渡しておけば、その都度ゼロから説明しなくても、いつでも“その道のプロ”として動いてくれます。

CLAUDE.md vs Skills

CLAUDE.md と Skills の使い分け

どちらも「Claude Code への指示を書くもの」に見えて混同しがちですが、役割ははっきり違います。いちばん大事な違いは 「いつ読むか」です。

CLAUDE.md

常に守る“全体ルール”

セッション開始時に毎回 自動で読む
  • プロジェクトに 1つ（ルートに置く）
  • 「コメントは日本語で」「本番DBは触らない」などの共通ルール
  • たとえると 就業規則・社内ルール

Skills

その時だけ使う“専門知識”

必要なタスクの時だけ 動的に読む
  • 複数OK（.claude/skills/ に置く）
  • 「LPを作る手順」「セキュリティ監査の観点」などの専門業務
  • たとえると 各専門家の業務マニュアル
観点 CLAUDE.md Skills
何を書くプロジェクト全体の業務ルール特定のタスク専用の指示パッケージ
いつ読むセッション開始時に必ず毎回必要な時だけ動的に
役割常に守るルールこの時だけ使う専門知識
設置場所プロジェクトルートのファイル1つ.claude/skills/ ディレクトリ
適用範囲プロジェクト全体呼び出した時だけ
たとえると社内規則・就業規則業務マニュアル・手順書（専門家）

CLAUDE.md は「何時に出社・退社」みたいな全社員共通のルール。
Skills は「経理の専門知識」「開発の専門知識」みたいな各専門家の業務知識。— 迷ったら “全員に常に効かせたいか／その業務の時だけ効かせたいか” で分けましょう。

How to Write

書き方：フォルダ構成と SKILL.md

Skills は .claude/skills/ の下に、スキルごとのフォルダを作って置きます。フォルダの中で必須なのは SKILL.md の1ファイルだけ。あとは任意です。

フォルダ構成 
.claude/skills/
└── your-skill-name/          # フォルダ名 = スラッシュコマンド名
    ├── SKILL.md              # 必須（大文字）
    ├── templates/            # 任意
    ├── examples/             # 任意
    └── references/           # 任意

SKILL.md の中身

スキルの発火条件（description）と、専門スキルの定義を書きます。フロントエンド特化、セキュリティ監査…などをここで定義します。

description は自動でOK

発火条件の description は Claude Code が自動で作ってくれるので、最初から細かく理解しておく必要はありません。

必須は SKILL.md だけ

skill.md や README.md ではダメで、大文字の SKILL.md が必要。templates/ 等は無くても動きます。

ポイントはたったこれだけ。.claude/skills/フォルダ名/SKILL.md を置けば、もう Skills の完成です。 最初はシンプルな1ファイルから始めて、必要になったら templates/ などを足していきましょう。

How to Call

呼び出し方は2つ

作った Skills は、次の2通りの方法で動かせます。

METHOD 1

自動起動

description の発火条件にもとづいて、Claude Code が自分で判断してスキルを起動します。話しかけるだけでOK。

「ランディングページをデザインして」と頼む
→ frontend-design スキルが自動で発動

この仕組みは「プログレッシブディスクロージャー」と呼ばれますが、用語は覚えなくて大丈夫です。

2METHOD 2

スラッシュコマンド（明示的に呼ぶ）

スキルを名前で直接呼び出します。以前の「カスタムスラッシュコマンド」が Skills に統合されました。

/frontend-design
/security-audit

フォルダ名がそのままコマンド名になります（例：frontend-design/ → /frontend-design）。

講座では基本スラッシュコマンドで呼び出します。 自動起動がうまく動かないときは、スラッシュコマンドで明示的に呼ぶほうが確実です。

Troubleshooting

よくあるつまずきと対処

EP11 の内容で引っかかりやすいポイントです。実演どおりに進まない時の参考にどうぞ。

CLAUDE.md と Skills の使い分けで毎回迷う

「日本語で書く」「コミットは英語」などの全体ルールは CLAUDE.md、「ランディングページを作る手順」のような専門業務は Skills。“常に効かせたいか／その時だけか”で分けます。

Skills が自動起動しない

description が曖昧だと発火条件を判定できません。説明を具体化する（「フロント開発時」→「React コンポーネント作成時」など）か、/skill-name で明示的に呼ぶのが確実です。

.claude/skills/ を作っても認識されない

フォルダ階層・ファイル名を確認。大文字の SKILL.md が必要（skill.md / README.md はNG）。配置後は Claude Code を再起動して読み込み直します。

/skill-name が「unknown command」になる

フォルダ名 = コマンド名です。一致しているか確認し、Ctrl+C ×2 → claude で再起動。プラグイン経由なら /Reload plugins も試します。
