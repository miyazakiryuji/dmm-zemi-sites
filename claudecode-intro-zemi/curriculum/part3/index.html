Web開発①：自分専用の秘書アプリを作って公開しよう（API不使用）

日本語で指示するだけで、自分専用の秘書アプリをゼロから作り、デザインや機能を整え、最終的にインターネット上へ公開するまでを体験します。外部APIは使わないので料金や難しい契約の心配はなく、Claude Code・GitHub・Cloudflare Pages を使った「作る→直す→自動で公開する」という現代的な開発の流れを、入門者でも一周できるように構成しています。

1

アプリ開発の基礎知識

まずは「アプリがどんなファイルでできているのか」をざっくりイメージしましょう。仕組みを完全に理解する必要はありません。完成形を先に見てから、その中身を軽く眺めるだけで十分です。

  • 今回作る秘書アプリは、ブラウザで動くシンプルなWebアプリです。外部のAPI（他社サービスとの連携）は使わないので、難しい契約や料金の心配なく作れます。
  • Webアプリは主に3種類のファイルでできています。骨組みを書くHTML、見た目を整えるCSS、動きをつけるJavaScriptの3つです。今回はこの組み合わせで作ります。
  • トップページのファイルは必ずindex.htmlという名前にします。公開サービスはこの名前を「入口」として自動で認識するためです。
  • メモやタスクなどのデータは、ブラウザ内の保存領域（localStorage）にためる作りにできます。サーバーやデータベースが不要なので、入門者でも完結しやすいのが利点です。
  • コードの中身を1行ずつ理解する必要はありません。「何を作りたいか」を日本語で伝えれば、Claude Code がファイルを用意してくれる——この感覚をつかむのがこのパートのゴールです。

手順

  1. 完成イメージを先に持つ：「メモを追加・削除できる」「やることリストを管理できる」など、欲しい機能を一言で書き出してみる。
  2. プロジェクト用のフォルダを1つ用意する（フォルダ名はsecretary-appのように半角英語にする）。
  3. そのフォルダをエディタ（Cursor など）で開き、ターミナルでclaudeと打って Claude Code を起動する。
  4. この後の手順で、Claude Code がindex.html・style.css・script.jsなどを自動で作ってくれることを確認する。

💡 補足・コツ

  • フォルダ名・ファイル名は半角英数字で。日本語名はトラブルの元になりやすいです。
  • 最初は仕組みが分からなくて当然です。「動くものを先に作る→あとで中身を眺める」の順番でOKです。
2

自分専用の秘書アプリを開発してみよう

いよいよ開発です。ごく簡単な日本語の指示だけで、Claude Code が秘書アプリを作ってくれます。入門者は「一気に全部任せる」より、提案を1つずつ承認しながら進めるのが安全です。

  • 指示はそうむずかしく考えなくて大丈夫です。「自分専用の秘書アプリを作って」と日本語で伝えるだけで、Claude Code が必要なファイルを提案してくれます。
  • Claude Code はファイルを作る前に「これを作っていいですか？」と確認してきます。入門者は1つずつ内容を見て承認するのがおすすめ。すべて自動で進める Auto Mode はこのパートの後半で紹介します。
  • 作りたい機能は具体的に伝えるほど良い結果になります。例：「メモを追加・削除できる」「やることリストにチェックを付けられる」「今日の日付を表示する」など。
  • 改行したいときはEnterではなくShift+Enter（またはAlt+Enter）を使います。Enter単独だと指示が送信されてしまいます。
  • 想定通り動かないのは普通のことです。あわてず、起きている状況をそのまま日本語で伝えれば、Claude Code が直してくれます。

手順

  1. Claude Code に作りたいものを伝える。例：「自分専用の秘書アプリをHTML・CSS・JavaScriptの3つのファイルで作ってください。メモとやることリストを管理できるようにしてください。」
  2. Claude Code が作るファイルの提案を1つずつ確認し、問題なければ承認して進める。
  3. ファイルができたら、index.htmlをブラウザで開いて見た目と動きを確認する（ファイルを右クリック→ブラウザで開く）。
  4. うまく表示・動作しない場合は、状況を伝えて修正してもらう（下の対処例を参照）。

💡 補足・コツ

  • うまく開けない・全部は動かないときの依頼例：「正常にアクセスできないので、全機能が正常に動作するように直してください」。
  • エラー画面が出たときの依頼例：「このエラーを修正してください」と書き、エラー画面のスクリーンショットや文章を一緒に貼り付けると、原因が特定されやすくなります。
  • 1回でうまくいかなくても落ち込まないこと。AIとの開発は「直して」「もう少しこうして」を何度か繰り返すのが普通の進め方です。
  • 承認を1つずつ行うのは「Claudeが何をしようとしているか」を見る練習にもなります。慣れてきたら範囲を広げていけば大丈夫です。
3

秘書アプリにデザインを当ててみよう

最初のアプリは機能は動くものの、見た目はやや素っ気ないことが多いです。ここでは日本語の指示だけで、見た目を一気に整えていきます。デザインの知識は不要です。

  • 「とりあえず作る」だけだと、いわゆる「AIっぽい少し地味なデザイン」になりがちです。ここから指示で印象を大きく変えられます。
  • デザインの指示はイメージを言葉で伝えるのがコツです。「落ち着いた色」「やさしい雰囲気」「スマホでも見やすく」など、感覚的な言葉でかまいません。
  • 色・余白・文字の大きさ・角の丸みなどをまとめて整えてもらえます。具体的な参考（例：好きなサービスの雰囲気）を伝えるとさらに精度が上がります。
  • スマホ表示への対応（レスポンシブ）も依頼できます。「スマホでも崩れず見やすくして」と一言添えるだけでOKです。
  • 気に入ったデザインの方向性（色やフォントの決まりごと）は、メモに残しておくと次回以降の指示が楽になります。

手順

  1. 見た目の希望を日本語で伝える。例：「全体のデザインを整えてください。落ち着いた配色で、余白を広めに、スマホでも見やすいレイアウトにしてください。」
  2. Claude Code がCSSを更新したら、ブラウザを再読み込みして変化を確認する。
  3. 気になる部分があれば部分的に修正を依頼する。例：「ボタンの色をもう少し優しいトーンにして」「見出しを大きくして」。
  4. スマホ表示を確認する。ブラウザの開発者ツール（MacはCmd+Option+I）でスマホ幅に切り替えてチェックする。

💡 補足・コツ

  • 「ダサい」「なんか違う」だけでも案外伝わりますが、「どう変えたいか」を一言添えると直しが早くなります（例：明るく／落ち着いた／余白多め）。
  • ブラウザに変更が反映されないときは、強制再読み込み（Mac: Cmd+Shift+R / Win: Ctrl+F5）を試してください。
4

秘書アプリに機能を追加してみよう

見た目が整ったら、便利な機能を足していきましょう。アプリは「少しずつ育てていく」もの。一度に全部ではなく、1機能ずつ追加するとトラブルを切り分けやすくなります。

  • 機能追加も日本語の指示でできます。例：「やることリストに『完了』チェックを付けられるようにして」「メモを後から編集できるようにして」。
  • 1回の指示で1機能ずつ追加するのがおすすめです。まとめて頼むより、動作確認しながら進める方が、どこで問題が起きたか分かりやすくなります。
  • API不使用でも実用的な機能は十分作れます。例：メモの並べ替え、検索、入力した内容の自動保存（ブラウザ内保存）、ダークモード切り替えなど。
  • 追加した機能が既存の動きを壊していないか、毎回ブラウザで軽く確認する習慣をつけると安心です。
  • うまくいかない機能は、状況を具体的に伝えて修正してもらいます。「追加したが動かない」より「保存ボタンを押しても一覧に出てこない」のように書くと直りが早いです。

手順

  1. 追加したい機能を1つ決めて伝える。例：「メモを保存できるようにして、ページを再読み込みしても消えないようにしてください。」
  2. Claude Code の変更を承認し、ブラウザで動作を確認する。
  3. 問題なければ次の機能へ。問題があれば状況を具体的に伝えて修正してもらう。
  4. ひと区切りついたら、現在の動きを一通り触ってみて「壊れていないか」を全体確認する。

💡 補足・コツ

  • 便利な「ブラウザ内保存」を頼むときは、「再読み込みしてもデータが消えないようにして」と伝えると意図が正確に届きます。
  • 機能を盛り込みすぎると複雑になります。まずは自分が毎日使う機能だけに絞ると、完成度が上がります。
5

秘書アプリをウェブで公開してみよう

手元（ローカル）で動くアプリを、インターネット上に公開してみましょう。今回は無料で手軽なCloudflare Pagesを使い、まずはファイルをアップロードする最もシンプルな方法で公開します。

  • 今のアプリは自分のパソコンの中でしか開けません。公開（デプロイ）すると専用のURLが発行され、誰でもアクセスできるようになります。
  • Cloudflare Pagesは、静的なWebサイト（HTML・CSS・JavaScript）を無料で公開できるホスティングサービスです。サーバーの契約や難しい設定は不要です。
  • 公開にはCloudflareの無料アカウントが必要です。メールアドレスで登録できます。
  • トップページのファイル名は必ずindex.htmlに。これがないと「ページが見つかりません」と表示されてしまいます。
  • 公開後にアプリを直したら、再度アップロードすれば内容が更新されます。次のトピック以降では、これをもっと自動化する方法も学びます。

手順

  1. Cloudflare の無料アカウントを作成し、ダッシュボードにログインする。
  2. メニューから「Workers & Pages」を開き、新しい Pages プロジェクトを作成する。
  3. 「Direct Upload（直接アップロード）」を選び、秘書アプリのフォルダ（index.htmlを含む）をアップロードする。
  4. アップロードが完了すると公開用のURL（https://〇〇.pages.dev）が発行される。ブラウザで開いて動作を確認する。
  5. 他の人にそのURLを共有して、ちゃんと開けるか確かめる。

💡 補足・コツ

  • .pages.devのURLはそのまま共有できます。独自のドメイン名を割り当てたい場合は、あとから設定で追加できます。
  • 公開後にCSSや画像が読み込まれない場合は、ファイルの読み込みパスが絶対パス（自分のPC上の場所）になっていないかを確認し、相対パスに直すよう Claude Code に依頼してください。
  • 公開するファイルに、自分だけのメモやテスト用のデータが残っていないか確認してから上げましょう。
6

GitHub とは

ここからは公開をもっと便利にするための土台、GitHub を学びます。最初は名前を聞いても何のことか分からなくて当然です。たとえ話で、その役割をやさしくつかみましょう。

  • GitHub（ギットハブ）は、ソースコードをインターネット上に保存・管理できるサービスです。エンジニアにとっての「クラウド保管庫＋作業履歴ノート」のような存在です。
  • 大きな役割は3つ。コードの保存（バックアップ）、バージョン管理（いつ・何を変えたかの履歴）、公開・共有（他のサービスとつなぐ土台）です。
  • バージョン管理とは、文章で言う「上書き保存の履歴が全部残る」イメージです。間違えても過去の状態に戻せるので安心して試せます。
  • コードを置く場所の単位をリポジトリ（repository）と呼びます。プロジェクト1つにつきリポジトリ1つ、と考えると分かりやすいです。
  • リポジトリには公開（Public）と非公開（Private）があります。Public は誰でも中身を見られ、Private は自分（と許可した人）だけが見られます。個人の作品は基本Privateが安全です。

💡 補足・コツ

  • 今は「コードを保存して、変更履歴を残し、公開サービスとつなぐための土台」とだけ覚えておけば十分です。
  • GitHub に置くと、Cloudflare Pages のような公開サービスと連携でき、更新が自動で反映される仕組みが作れます（このパートの最後で扱います）。

Claude Code にルールを設定してみよう

GitHub に公開する前に、必ず通っておきたい「安全の関所」がルール設定です。Claude Code にはRules（ルール）を設定でき、特にセキュリティのルールはコードを外に出す前に固めておきます。

  • Claude Code は、プロジェクト内のCLAUDE.mdというファイルに書いたルールを毎回読んでくれます。これがRules（ルール）の正体で、「毎回説明しなくていい」状態を作れます。
  • GitHub にコードを上げる前に、セキュリティのルールをテンプレとして設定しておくのが重要です。うっかり秘密情報を公開してしまう事故を防げます。
  • .gitignoreというファイルに名前を書いておくと、そのファイルはGitHubにアップロードされません。.env（秘密情報を入れるファイル）などをここで除外します。
  • .envにはAPIキーやパスワードなどの秘密情報を入れます（今回のアプリでは使いませんが、扱い方は今のうちに覚えておきましょう）。絶対に公開してはいけません。
  • ルールは具体的に書くのがコツです。「気をつけて」ではなく「.envは絶対に編集・公開しないこと」「秘密情報をコードに直接書かないこと」のように明確に書きます。

手順

  1. Claude Code で/initコマンドを実行し、プロジェクトを分析したCLAUDE.mdを自動生成する。
  2. 生成されたCLAUDE.mdに、セキュリティのルールを追記してもらう。例：「.envやAPIキーは絶対に公開しないルール、.gitignoreの使い方をCLAUDE.mdに追記してください。」
  3. .gitignoreに.envや.env.*などの秘密情報ファイルが含まれているか確認する。
  4. ルールを更新したら、Claude Code を一度終了（Ctrl+Cを2回）してclaudeで再起動し、新しいルールを読み込ませる。

💡 補足・コツ

  • 厳格なセキュリティのルール例を1つ作っておくと、次のプロジェクトでも使い回せて便利です。「秘密情報は.envに置く」「.envは.gitignoreで除外する」「公開前に秘密情報の混入を確認する」をセットで覚えましょう。
  • 重要なルールはCLAUDE.mdの冒頭に置くと、より守られやすくなります。
  • 万一.envやAPIキーを公開してしまったら、そのキーは流出した前提で即座に無効化し、新しいキーを発行してください。
8

GitHub の使い方

ここで実際に GitHub のアカウントを作り、基本操作に触れます。難しい設定はなく、Googleアカウントなどで数分で登録できます。

  • 公式サイトから無料アカウントを作成します。Googleアカウントでの登録が一番スムーズです。
  • 登録時にユーザーネームを決めます。これは世界共通のIDで、自分のページのURLにもなります。覚えやすく、ググりやすい名前にしましょう。
  • 登録の最後に、ロボットでないことを確認する画像認証（CAPTCHA）があります。指示に従ってパズルを解けば完了です。
  • ログイン後の「Repositories」タブが、自分のコードを保管していく場所です。プロジェクトごとにリポジトリが並びます。
  • 個人の作品はPrivate（非公開）で作るのが基本です。公開が必要になったら、あとから設定で切り替えられます。

手順

  1. GitHub公式サイトにアクセスし、「Sign up（登録）」を開く。
  2. Googleアカウント、またはメールアドレスとパスワードで登録する。
  3. ユーザーネームを決め、国（Japan）を選んで「Create a free account」をクリックする。
  4. 画像認証（CAPTCHA）を完了し、確認メールが届いたら認証してアカウント作成を終える。
  5. ログイン後、プロフィールの「Repositories」タブを開いて、コードの保管場所を確認する。

💡 補足・コツ

  • ユーザーネームが「既に使われています」と出たら、数字やハイフンを足して差別化しましょう（例：tanaka-dev）。
  • 画像認証が何度も通らないときは、VPNをオフにする・別のブラウザで試す・シークレットウィンドウで開く、を試してください。

GitHub にソースコードをプッシュしてみよう

作った秘書アプリのコードを GitHub に保存（プッシュ）します。難しいコマンドは Claude Code に任せられますが、最初の「認証」だけは自分の手で行う場面があります。

  • コードを GitHub に送ることをプッシュ（push）と呼びます。これでクラウドに保存され、変更履歴も残るようになります。
  • プッシュの前に、自分のパソコンと GitHub を認証（連携）でつなぎます。GitHub CLIという公式ツールを使うのが一番簡単です。
  • 認証コマンドgh auth loginは対話形式（質問に答える形）のため、Claude Code から直接実行するとうまくいきません。コマンドの先頭に!を付けて実行します（!gh auth login）。!は「ターミナルで直接実行」の合図です。
  • 認証が済めば、あとは日本語の依頼でプッシュできます。「この秘書アプリを GitHub のリポジトリにアップロードしてください」と伝えるだけです。
  • アップロード前に必ず.gitignoreを確認し、秘密情報（.envなど）が一緒に上がらないことを確かめます。これがトピック7のルール設定が効いてくる場面です。

手順

  1. GitHub CLI をインストールし、エディタを再起動してからgh --versionで動作を確認する。
  2. Claude Code に「GitHubアカウントを認証してください」と依頼すると、!gh auth loginを案内されるので、それをターミナルで実行する。
  3. 表示されたリンクを開いてGitHubにログインし、画面のコードを貼り付けて「Authorize（認証）」する。
  4. Claude Code に「この秘書アプリをPrivateでGitHubのリポジトリにアップロードしてください」と依頼する。
  5. GitHubのリポジトリページを開き、コードが上がっていること、.envなどの秘密情報が含まれていないことを確認する。

💡 補足・コツ

  • プッシュ時は「Privateで」と明示しましょう。何も言わないと公開（Public）で作られることがあります。
  • もし.envを誤って上げてしまったら、まず.gitignoreに追加し、Claude Code に履歴からの削除を依頼。該当のAPIキーは無効化して作り直すのが鉄則です。
  • !付きで実行するのは「認証など対話が必要なコマンド」のときだけ、と覚えておけば十分です。
10

アプリのウェブ公開を自動化してみよう

最後に、GitHub と Cloudflare Pages をつないで「公開の自動化」を体験します。これができると、コードを直して GitHub に送るだけで、公開中のアプリが自動で最新版に更新されます。

  • トピック5では手動でファイルをアップロードしましたが、ここではGitHub と Cloudflare Pages を連携させます。手動アップロードはもう不要になります。
  • 一度連携すれば、コードを更新して GitHub にプッシュするだけで、Cloudflare Pages が自動で受け取り、公開サイトを更新してくれます。
  • この「保存したら自動で公開される」流れは、プロのソフトウェア開発でも標準的な仕組みです。難しい設定なしに、その入口を体験できます。
  • 連携時に「どのリポジトリを使うか」を選ぶだけで、ビルド（公開準備）の設定はシンプルな静的サイトならほぼ初期値のままでOKです。
  • 公開URLは引き続きhttps://〇〇.pages.dev。中身が変わってもURLは変わらないので、共有先に再連絡する手間もありません。

手順

  1. Cloudflare のダッシュボードで「Workers & Pages」から新しい Pages プロジェクトを作成し、今度は「Connect to Git（GitHub連携）」を選ぶ。
  2. GitHubアカウントへのアクセスを許可し、秘書アプリのリポジトリを選択する。
  3. ビルド設定を確認する（HTML・CSS・JavaScriptだけの静的サイトなら、特別なビルドコマンドは不要なことが多い）。「Save and Deploy」で公開する。
  4. 発行されたURLでアプリが表示されることを確認する。
  5. 試しにアプリを少し直し、GitHubへプッシュして、しばらく待つと公開サイトが自動で更新されることを確認する。

💡 補足・コツ

  • 更新がすぐ反映されないように見えるときは、少し待ってから強制再読み込み（Mac: Cmd+Shift+R）を試してください。反映には少し時間がかかることがあります。
  • もし公開がうまくいかない（白い画面・崩れなど）ときは、Cloudflareの「デプロイ（Deployments）」のログを見て、その内容を Claude Code に貼り付けて相談しましょう。
  • ここまで来れば、「直す→GitHubに送る→自動で公開される」という、現代的な開発の一連の流れを一周したことになります。自分だけの秘書アプリを、これからも気軽に育てていけます。
